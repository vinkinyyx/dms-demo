## 四、发现的问题与修复建议

以下问题在编写测试与阅读业务代码时发现，**未修改业务源码**，仅记录以便后续处理。

### P1 - 阻塞类

| # | 问题 | 位置 | 影响 | 建议 |
|---|------|------|------|------|
| 1 | `InventoryRepository.lockKeyed` 使用了 PostgreSQL 特有的 `IS NOT DISTINCT FROM` 语法（native SQL） | `backend/src/main/java/com/dms/inventory/repository/InventoryRepository.java` | H2 兼容模式下语法可能报错，导致部分库存集成测试在 H2 环境失败；生产不受影响 | 方案 A：为测试环境提供 H2 版本的 Repository（`@Profile("test")`）；方案 B：在 Repository 中用 `(:x IS NULL AND col IS NULL) OR col = :x` 等 SQL 通用表达式代替 |

### P2 - 一般

| # | 问题 | 位置 | 影响 | 建议 |
|---|------|------|------|------|
| 2 | `AuthService.locateUser` 未指定 tenantCode 时会 `userRepository.findAll()` 后内存过滤 | `AuthService.java:177` | 用户量大时性能急剧下降；且 tenantCode 空的情况下 `findAll()` 未按 tenant 隔离，存在越权风险 | 应改为按 `username` 全局唯一索引查询，或强制要求 tenantCode |
| 3 | `AuthService.login` 密码错误分支调用 `userService.incrementFailCount`，但异常抛出后事务标记 rollback，可能导致失败计数不生效 | `AuthService.java:65-66` | 登录失败次数无法累计，无法触发锁定 | 将 `incrementFailCount` 改为独立事务（`REQUIRES_NEW`） |
| 4 | `PromotionEngine` 未处理 `FULL_REDUCTION` 与 `MOQ` 同时 exclusive 的跨类型互斥；`exclusive` 仅同类型内互斥 | `PromotionEngine.java:66-78` | 若产品目标是跨类型互斥，会双重生效 | 根据实际业务需求确认互斥语义并调整实现 |
| 5 | `WechatMockService.handleCallback` 强依赖 code 以 `MOCK_OPENID_` 开头，接口未做长度上限校验 | `WechatMockService.java:74` | 生产环境切换真实微信时需重构，测试时也易误拼 | 在切换真实实现前抽出 `WechatClient` 接口 |
| 6 | `ForgotPasswordRequest`/`ResetPasswordRequest.reset` 属于占位实现，直接 log.info 无实际动作 | `AuthService.java:100-110` | 用户故事 US-2.7 尚未真正实现，只是接口通路存在 | V1 显式声明为"管理员通道"，或加临时 rate limit 保护 |

### P3 - 建议

| # | 问题 | 建议 |
|---|------|------|
| 7 | 项目根目录未提供 `mvnw`/`mvnw.cmd` | 建议 `mvn -N wrapper:wrapper` 生成 wrapper，方便无 Maven 环境下运行 |
| 8 | `application-test.yml` 使用 H2 但业务大量依赖 PostgreSQL JSONB / partition table / native SQL | 建议引入 Testcontainers + PostgreSQL 15 做真集成测试，提升可靠性 |
| 9 | 缺少 CI 集成 | 建议增加 `.github/workflows/backend-test.yml`，PR 阻塞 |
| 10 | `PromotionEngine.evaluate` 对 `lines==null` 与 `tenantId==null` 早退，但未记录日志 | 添加 debug 日志便于排查 |

### 5.3 假设与约束记录

1. 假设 `dms.jwt.secret` 长度足够（至少 256 bit for HS256）；测试中使用 `"unit-test-secret-key-abcdefghijklmnopqrstuvwxyz-1234567890"`（>= 48 字节）满足要求
2. 假设 H2 `MODE=PostgreSQL` 能覆盖 90% 的 JPA 场景；对于 native SQL 中 PostgreSQL 特有语法（IS NOT DISTINCT FROM、JSONB 操作），已列入问题清单
3. 假设集成测试使用 `@Transactional` 每个测试后回滚，因此测试之间无数据污染
4. Redisson/Redis 通过 `@MockBean` 打桩 + Spring autoconfig exclude 两种方式确保不依赖外部 Redis 实例
