## 三、五大改造点详解

### ① 状态驱动按钮（问题 1）

**销售订单**：
| 状态 | 允许操作 |
|---|---|
| DRAFT 草稿 | 查看 · 编辑 · 提交审批 · 取消 |
| SUBMITTED 已提交 | 查看 · 审批通过 · 驳回 · 取消 |
| APPROVED 已审批 | 查看 · 出库发货 |
| COMPLETED / REJECTED / CANCELLED | 仅"查看" |

**采购订单**（更完整）：
| 状态 | 允许操作 |
|---|---|
| DRAFT | 查看 · 编辑 · 提交审批 · 取消 |
| SUBMITTED | 查看 · 审批通过 · 驳回 · 取消 |
| APPROVED | 查看 · **📦 收货入库** |
| RECEIVING 收货中 | 查看 · 📦 继续收货 |
| COMPLETED | 仅"查看" |

实现方式：前端 `actionsForStatus(menu, status)` 根据 status 返回不同按钮数组，服务端 `PurchaseOrderController.allowedActions()` 与详情一起返回。

### ③ 采购销售彻底拆分（问题 3）

**菜单结构变化**：

之前：
```
📁 订单与销售
  ├─ 📋 订单管理（混在一起）
  ├─ 🛒 销售出库
  ├─ 💵 采购发票
  └─ 💰 销售发票
```

现在：
```
📁 销售业务
  ├─ 📤 销售订单
  ├─ 🛒 销售出库
  └─ 💰 销售发票

📁 采购业务
  ├─ 📥 采购订单 ⭐
  ├─ 📦 收货入库
  └─ 💵 采购发票
```

**数据库拆分**：
- `orders` + `order_lines` （销售用，字段 `dealer_id`）
- `purchase_orders` + `purchase_order_lines` （采购用，字段 `supplier_id` + `warehouse_id`）

**下单页统一但双模式**：
- `/order-create.html?mode=sales` → 显示"经销商" + 授信额度
- `/order-create.html?mode=purchase` → 显示"供应商" + "入库仓库"字段

### ④ 库存自动联动（问题 4）

**采购收货入库流程**：

```
1. 创建采购单：数量 50
   └─ purchase_orders INSERT

2. 提交审批
   └─ status: DRAFT → SUBMITTED

3. 审批通过
   └─ status: SUBMITTED → APPROVED

4. 点击【📦 收货入库】
   ├─ purchase_order_lines.received_qty += 50
   ├─ inventory (product_id, warehouse_id) 存在? UPDATE qty += 50 : INSERT
   ├─ inventory_transactions INSERT (RECEIPT, +50, ref_doc=purchase_order/N)
   └─ 检查所有明细是否收完 → status: APPROVED → COMPLETED
```

**已通过冒烟测试验证**：
```
库存 150 → 收货 50 → 库存 200 ✅（自动 +50）
```

## 五、关键 API 一览

### 采购订单
```
GET    /api/purchase-orders                       列表（支持 status、supplierId 过滤）
GET    /api/purchase-orders/{id}                  详情（含明细 + allowedActions）
POST   /api/purchase-orders                       创建
PUT    /api/purchase-orders/{id}                  编辑（仅 DRAFT）
POST   /api/purchase-orders/{id}/submit           提交审批
POST   /api/purchase-orders/{id}/approve          审批通过
POST   /api/purchase-orders/{id}/reject           驳回
POST   /api/purchase-orders/{id}/cancel           取消
POST   /api/purchase-orders/{id}/receive          收货入库 ⭐
```

### 销售订单元数据
```
GET    /api/orders/{id}/allowed-actions           按 ID 查询允许的操作
GET    /api/orders/actions-for-status?status=X    按状态字符串查询允许的操作
```

### 低代码字段配置
```
GET    /api/form-configs/forms                    表单列表
GET    /api/form-configs/{formKey}                查询该表单的字段配置
POST   /api/form-configs/{formKey}/upsert         批量新增/更新字段
DELETE /api/form-configs/{id}                     删除字段配置
```

### 字典管理
```
GET    /api/dicts/types                           字典分类列表
POST   /api/dicts/types                           新增字典分类
GET    /api/dicts/{typeCode}/items                字典条目
POST   /api/dicts/{typeCode}/items                新增条目
PUT    /api/dicts/items/{id}                      更新条目
DELETE /api/dicts/items/{id}                      删除条目
```

### 库存汇总
```
GET    /api/inventory-summary/by-product/{id}     产品实时库存（总量/临期/呆滞/按仓库）
GET    /api/inventory-summary/dealer-overview/{id}  经销商概览（订单/授信）
```

## 六、修复过程遇到的坑（10 个）

| # | 问题 | 解决方案 |
|---|---|---|
| 1 | Hibernate native SQL `?N::jsonb` 语法冲突 | 改为命名参数 + `CAST(:x AS jsonb)` |
| 2 | 一条 SQL 里 `?7` 位置参数被引用两次 | 拆成 `?7` 和 `?8` 独立参数 |
| 3 | 服务器磁盘 94% 满 Postgres 挂掉 | 清理 12 GB 无关缓存 |
| 4 | `inventory_transactions` 实际列名是 `qty_change/ref_doc_id/ref_doc_type` | 对齐迁移 V4 的定义 |
| 5 | `purchase_invoices` 实际列名是 `invoice_date`，无 status | 修 SQL |
| 6 | `data_scopes` 无 `target_id` 列，是 `dealer_ids/org_ids` | 修 SQL |
| 7 | `workflows` 无 `description` 列 | 修 SQL |
| 8 | `workflow_nodes` 列名 `node_code` 实际是 `code` | 修 SQL |
| 9 | `loans` 表列 `lender_dealer_id / borrower_dealer_id` | 修 SQL |
| 10 | 事务级联失败 | 每个查询 `@Transactional(propagation=REQUIRES_NEW)` |
