**1. 需求背景**

**1.1 业务背景**

在快消、零售、医疗器械等轻资产分销行业，厂商通常通过多层级经销商网络（一级/二级/三级）触达终端市场。厂商需要对分布在全国的数百至数千家经销商进行合同签订、授权维护、订单收发、库存监控、销售上报、返利结算、审计合规等全生命周期管理。传统模式依赖线下签约、Excel 台账、邮件对账，普遍存在如下问题：

- 渠道数据滞后：经销商销售、库存、动销数据回传延迟 5–15 天，厂商无法实时看清市场；

- 合同流程冗长：新建/变更/续约/终止均线下走单，平均耗时 2–4 周，且难以留痕；

- 主数据不一致：产品、价格、医院/终端、区域等主数据在 ERP、CRM、销售部各持一份，口径打架；

- 授权难管控：区域串货、越权销售、超期销售等违规行为难以事前拦截；

- 审计合规压力大：医疗器械/食药类产品需支持流向追溯（UDI/批号/序列号），传统台账无法满足；

- 返利/结算不透明：达成、返点、折扣计算规则不透明，经销商与厂商对账争议多。

**1.2 建设目标**

通过一体化的 DMS 平台，将合同、授权、订单、库存、销售、返利、报表、审计等业务线上化、数据化，形成"厂商-经销商-终端"的数据闭环，具体目标如下：

- 渠道数据 T+1 回传，Sales-Out 数据实时对齐 Sales-In，形成 IMS 视图；

- 合同全生命周期线上化，审批周期从 2–4 周压缩到 3–5 个工作日；

- 主数据统一由厂商侧维护，经销商侧只读，杜绝主数据打架；

- 通过授权配置对订单、销售、退换货、借货等场景做事前控制，减少违规；

- 全链路批号/序列号追溯，满足医疗器械 UDI、快消召回、生鲜溯源等合规要求；

- 返利、指标、达成率自动计算，规则前置可视，经销商侧可自助查询。

**1.3 目标用户与规模**

平台面向轻资产分销行业的三类主体，典型规模如下：

|  |  |  |
|----|----|----|
| **角色** | **典型职责** | **典型并发规模** |
| 厂商管理员 | 系统配置、组织权限、主数据维护、日志审计 | 5–20 人 |
| 厂商内部业务用户 | 销售/运营/客服/财务/合同专员/供应链 | 50–500 人 |
| 经销商用户 | 一/二/三级经销商的下单、库存、销售、报台、对账 | 500–5000 家 × 每家 2–20 人 |

**2. 需求目标**

**2.1 核心度量目标**

|                          |                  |                              |
|--------------------------|------------------|------------------------------|
| **指标**                 | **现状（典型）** | **上线后目标**               |
| 经销商月末库存上报及时率 | 60% – 70%        | ≥ 95%（每月最后 3 个工作日） |
| 经销商销售上报及时率     | 70% – 80%        | ≥ 95%（次月第 5 个工作日前） |
| 合同审批平均周期         | 2 – 4 周         | ≤ 5 个工作日                 |
| 订单从下单到发货平均时长 | 3 – 5 天         | ≤ 2 天                       |
| 主数据不一致工单         | 月均 20 单+      | ≤ 3 单/月                    |
| 退换货处理时长           | 2 – 3 周         | ≤ 7 天                       |

**2.2 边界（非目标）**

- 本平台不替代 ERP，不承担财务总账、税务、供应链计划等能力，仅通过接口与 ERP/WMS/HR/OA 联动；

- 本平台不承担经销商内部的门店 POS、导购管理等 C 端业务；

- 本平台不做 BI 建模，仅提供固定报表 + 经销商画像；深度分析交由 BI/数仓完成。

**4. 用户角色**

平台采用"厂商侧 + 经销商侧"双域架构，每一域内按业务职责细分角色，权限基于 RBAC 四层模型（组织 → 资源 → 策略 → 角色）配置。

**4.1 厂商管理员**

负责整个 DMS 平台的运维配置，是权限体系的最终授予者。核心场景包括：搭建组织架构树、定义系统资源（菜单/按钮/API）、组合策略、创建角色、创建用户并挂角色、维护字典表、监控主数据缓存刷新、查询各类审计日志。该角色不参与日常业务操作，但对全平台数据具有只读可见性以便定位问题。

**4.2 厂商内部业务用户**

包括销售、销售经理、客服（CS）、商务经理、合同专员、供应链、财务、法务、审计、渠道管理等子角色。核心场景为日常业务处理：

- 销售/销售经理：发起合同申请、退货授权、临时授权，跟踪指标达成；

- 客服 / 商务：审批订单、审批库存调整、审批退换货、处理拆单；

- 合同专员：生成协议、维护模板、盖章、扫描回传、法务归档；

- 供应链：确认发货、处理缺货订单、维护发货信息；

- 财务：核对返利、发票登记、审计导出；

- 渠道管理员：维护经销商主数据、指标、区域-销售岗位映射。

**4.3 经销商用户**

分为经销商管理员（\`xxx_admin\`）与经销商操作员：

- 经销商管理员：管理本经销商下的子账号（新增、重置密码、启停用），维护本经销商基本信息；

- 经销商操作员：日常下单、收货、库存管理、销售出库、报台、发票上传、对账查询。

经销商用户仅能看到与本经销商相关的数据，通过"数据权限 = 经销商 ID"实现强隔离；一级经销商可查看其下挂二级经销商的分销出入库数据（受授权控制）。

**6.A 合同管理子系统 (DCMS)**

**6.A.1 主数据管理**

**模块概述**

维护合同签约所依赖的基础数据：产品档案、产品分类树、组织架构、行政区域（省/市/区）、终端客户（医疗行业为医院、快消行业为门店/连锁客户）、合同负责人（销售/商务归属）。主数据一经启用即被授权、订单、发票等下游模块引用，因此新增、变更、停用均需严格审批与版本化。厂商内部渠道管理员为主要操作者，主数据支持从 ERP 单向同步与手工维护两种模式，冲突时以 ERP 数据为准并回写标记。

**页面布局**

- 顶部条件区：编号/名称模糊查询、状态筛选、所属分类；

- 中部数据网格：分页列表，支持列排序、多选、批量导入/导出；

- 右侧操作栏：查看、编辑、启用/停用、关联关系维护；

- 侧滑弹窗：用于关联对象选择（如"大区 ↔ 省份"、"销售岗位 ↔ 终端"）。

**业务逻辑**

用户在数据网格中执行 CRUD 操作。新增时系统按数据类型自动生成编号（可覆盖），并校验唯一性；编辑时对被下游引用的字段（如产品编码、终端编码）禁止修改，只能停用后新增替代记录；停用时系统检查是否存在未关闭的授权、订单、库存，若存在则阻止停用并给出关联清单。批量导入采用"模板下载 → 填写 → 上传 → 校验 → 导入"四步流程，校验失败的记录在 Grid 中红色标注并给出原因，允许用户在页内二次修正后重新提交。任何变更均写入审计日志，包含操作人、时间、前值、后值。

**6.A.2 合同申请**

**模块概述**

覆盖经销商的全生命周期：新建、变更、续约、终止、批量延展、批量更新。销售或渠道管理员发起申请后，系统按合同分类自动提示所需上传资质（营业执照、经营许可证、GSP/GMP、法人身份证等），并对关键字段做前置校验；变更申请需列出变更前后对照（表格内容变更与非表格内容变更分开展示），便于审批人快速识别差异。

**业务逻辑**

用户点击「新建申请」进入表单页，逐步填写：合同类型（厂商-LP / 厂商-LS / 厂商-T1 / LP-T2 / LS-BSA / 非正式授权）→ 经销商基础信息 → 授权信息（终端、产品线、区域、有效期）→ 指标信息（年度采购额、Sales-Out 目标、达成分档）→ 上传附件 → 提交。变更申请从既有生效合同复制信息作为基线，允许用户修改字段，系统自动生成"变更前 vs 变更后"对照视图。续约申请支持一键复制历史合同并批量修改到期日；批量延展/批量更新支持一次维护多个合同的指标或到期日。终止申请提交后自动触发授权终止规则（见 6.B.2）。

**6.A.3 合同审批**

**模块概述**

提供 Web 端审批中心与邮件审批双通道。审批流由「流程设定」模块预先按合同分类、金额区间、授权区域等维度配置，运行时动态匹配审批人；不同节点向审批人展示的字段可差异化配置（如财务节点只看金额相关字段，法务节点只看合同条款）。

**业务逻辑**

审批中心以待办列表为入口，支持按类型、发起人、时间筛选。用户点击进入审批详情页，查看合同全字段、变更对照、附件预览；页面底部提供「通过」「拒绝」「转交」「加签」按钮。拒绝时必须填写原因，原因回写至发起人。邮件审批：系统在有新待办时向审批人邮箱发送带一次性 Token 的链接，点击后直达审批详情页无需登录（Token 24 小时失效）；审批结果同步回 Web 端。所有审批动作写入审计日志。

**边界与异常**

- 审批人离职：由管理员在流程设定中调整默认审批人；未处理的单据自动转派

- 并发审批：同一单据在两个节点被同时打开时，先提交者生效，后提交者获得冲突提示；

- 超时未审批：24/48/72 小时未处理分别发送提醒、加急、上升到上级。

**6.A.4 电子协议与签章**

**模块概述**

审批通过后自动生成电子协议 PDF，按「合同分类 × 经销商类型 × 申请类型」使用不同模板，模板中的动态字段（经销商名称、授权终端、有效期等）自动填充。对于需要特殊条款的经销商，支持"特殊条款批注"上传后走额外审批；不支持电子签章的场景（如线下签约），允许上传纸质协议扫描件替代。签章采用第三方 CA 接口，双方通过短信验证码/UKey 完成身份认证后签章。

**业务逻辑**

审批通过后系统在 5 分钟内生成协议 PDF 并通知双方。经销商侧登录后在「待签章」列表中看到该协议，点击进入预览页 → 阅读协议 → 输入短信验证码 → 完成经销商签章。厂商侧合同专员随后在「待厂商签章」列表中进入相同流程完成厂商签章。任意一方拒绝签章则回退到发起人，允许其重新发起变更或作废。签章完成后 PDF 落章版本入库，同步至法务归档，并触发 ERP 主数据同步与授权生成。

**字段规则**

|                |          |          |                  |
|----------------|----------|----------|------------------|
| **字段**       | **类型** | **必填** | **说明**         |
| 协议编号       | 文本     | 是       | 自动生成         |
| 协议模板       | 字典     | 是       | 按合同分类匹配   |
| 经销商签章时间 | 日期时间 | 否       | 经销商完成后写入 |
| 厂商签章时间   | 日期时间 | 否       | 厂商完成后写入   |
| 特殊条款附件   | 文件     | 否       | ≤ 10MB PDF       |
| 纸质扫描件     | 文件     | 否       | ≤ 20MB PDF/JPG   |
| CA 流水号      | 文本     | 否       | 第三方 CA 回写   |

**6.A.5 外部对接与法务归档**

电子协议生效后按接口规范推送至 ERP：新增经销商则新增 ERP 客户档案，变更/续约则更新档案，终止则冻结档案。所有生效协议 PDF 归档至法务系统（或本地归档目录），保留期不少于合同到期后 5 年；非电子签章的合同支持扫描件回传归档。归档记录包含合同编号、经销商编号、生效日期、终止日期、附件路径、审计签名，供审计人员随时导出。

**6.B 进销存子系统 (DMS)**

**6.B.1 主数据同步**

进销存所需主数据来自合同子系统（经销商、授权）、ERP（产品、价格、发票、发货）、HR（销售架构、销售指标）。系统通过接口在每日凌晨全量同步 + 实时增量同步双策略保证数据一致性。主数据在进销存子系统中不可编辑，只能查看；对于需要业务侧扩展的字段（如产品别名、包装规格），单独存储在进销存的"扩展表"中并与主数据关联。上线前需完成"期初库存导入"作为经销商首次使用的库存基线。

**字段规则（经销商授权同步表）**

|               |          |          |                                            |
|---------------|----------|----------|--------------------------------------------|
| **字段**      | **类型** | **必填** | **说明**                                   |
| 经销商编号    | 文本     | 是       | 合同子系统主键                             |
| 授权类型      | 字典     | 是       | 订单/销售到医院/销售到下级/退换货/移库借货 |
| 授权对象      | 文本     | 是       | 产品线 / 终端 / 区域 / 下级经销商          |
| 有效期起 / 止 | 日期     | 是       | 按授权类型分别配置                         |
| 状态          | 字典     | 是       | 生效/失效/暂停                             |

**6.B.2 授权管理**

**模块概述**

授权是所有交易的准入门槛，包含"合同授权"（由合同子系统下发，不可编辑）与"临时授权"（可在页面申请-审批后生效）。授权控制粒度可精确到「经销商 × 产品线 × 终端 × 授权类型」组合，且不同授权类型（订单/销售/退换货/移库/借货）可设置不同的有效期，实现事前控制。

**业务逻辑**

订单、销售出库、退换货、移库、借货等业务单据在提交时，系统按当前登录经销商与单据明细逐条校验授权：若无匹配授权或授权已过期则阻止提交并给出精确到"哪个产品/哪个终端/哪种类型"的失败原因。临时授权申请由销售或经销商发起，填写授权类型、对象、有效期、原因、附件后提交，商务审批通过后临时授权立即生效并计入授权控制；临时授权到期自动失效。授权终止后系统保留一段"窗口期"（默认 30 天，可配置）供经销商完成尾单的订单/销售/退换货/移库/借货操作，窗口期后彻底封锁。

**状态流转**

|              |              |              |              |
|--------------|--------------|--------------|--------------|
| **当前状态** | **允许操作** | **下一状态** | **触发条件** |
| 申请中       | 通过 / 拒绝  | 生效 / 拒绝  | 审批完成     |
| 生效         | 暂停 / 到期  | 暂停 / 失效  | 手动或时间   |
| 暂停         | 恢复 / 到期  | 生效 / 失效  | 手动或时间   |
| 失效         | （无）       | —            | —            |

**6.B.3 订单管理**

**模块概述**

经销商向厂商采购的核心链路。支持三类订单：采购订单（正常采购）、缺货订单（因厂商缺货预留下单资格）、定制订单（客户化需求）。订单从下单到关闭全程可追踪，覆盖 10+ 状态节点。

**页面布局**

- 顶部：订单号、状态、类型、提交日期等条件筛选；

- 中部：订单列表（分页 + 排序），行内展示关键字段与操作按钮；

- 右侧滑出：订单详情，含订单主信息、明细行、收货地址、附件、审批意见、发货信息；

- 顶部快捷入口：新建订单、批量导入、导出。

**业务逻辑**

经销商用户点击"新建订单"，选择订单类型 → 系统按授权列出可下单产品 → 用户添加产品与数量 → 系统按产品价格主数据自动带入单价与折扣 → 选择/新增收货地址 → 填写联系人/备注 → 保存草稿 或 提交。提交后进入厂商客服审批：客服可通过、拒绝、拆单（同一订单拆为多子单）或退回修改；通过后订单进入 ERP，ERP 完成审批 → 备货 → 发货，各节点状态实时回写。发货生成后系统自动为经销商生成待收货单据（见 6.B.4）；经销商确认收货后订单进入"已发货 → 已确认"，上传发票后订单关闭。经销商可撤销自己的草稿/已提交订单（未经审批）、修改已被退回订单、对已发货订单发起售后。

**字段规则**

|                          |          |          |                                |
|--------------------------|----------|----------|--------------------------------|
| **字段**                 | **类型** | **必填** | **说明**                       |
| 订单号                   | 文本     | 是       | 自动生成                       |
| 订单类型                 | 字典     | 是       | 采购/缺货/定制                 |
| 产品线 / 产品编码 / 数量 | 组合     | 是       | 数量为正整数，需满足最小起订量 |
| 收货地址                 | 结构     | 是       | 省/市/区/详细地址/联系人/电话  |
| 备注                     | 文本     | 否       | ≤ 500 字符                     |
| 期望到货日               | 日期     | 否       | 不早于今日 + 3                 |
| 折扣                     | 数值     | 否       | 0–100%，来源于合同或活动政策   |

**状态流转**

|  |  |  |  |
|----|----|----|----|
| **当前状态** | **允许操作** | **下一状态** | **触发条件** |
| 草稿 | 编辑/删除/提交 | 已提交待审批 / — | 发起人操作 |
| 已提交待审批 | 撤销/修改/审批 | 已撤销待审批/等待经销商修改/审批通过/审批拒绝 | 双方操作 |
| 审批通过 | 进入 ERP | 已进入 ERP 系统审批 | 接口推送 |
| 已进入 ERP 系统审批 | ERP 反馈 | ERP 已反馈 / ERP 已关闭订单 | ERP 回写 |
| ERP 已反馈 | ERP 发货 | ERP 已发货 / 发货中 | ERP 回写 |
| ERP 已发货 | 经销商收货 | 已确认 | 经销商确认收货 |
| 已确认 | 上传发票 | 已完成 | 发票上传/确认 |
| 任何状态 | 客户端异常 | 已拒绝 / 已完成 | 客服/ERP 处理 |

**边界与异常**

- 库存不足：客服可选择"整单拒绝"或"部分拆单"（未拆部分转缺货订单）；

- 价格失效：下单时的价格锁定，若审批期间价格变更，仍以下单价为准；超差异阈值需上级复核；

- 批量导入失败：错误行在 Grid 中标红，允许原地修正并二次提交；

- 并发：客服在审批时，若经销商同时修改订单，系统提示"订单已被发起人修改"并强制刷新。

**6.B.4 收货管理**

**模块概述**

接收厂商发货、上级经销商分销出库、其他经销商借货三类入库来源。发货方一旦完成出库，系统自动为收货方生成"待收货"单据；收货方可整单收货，也可通过扫描条码/序列号做逐件核对（医疗器械行业强制序列号核对）。收货前物品处于"在途库"，不进入可用库存；确认收货后按明细行拆入指定仓库并写入批号、序列号、生产日期、有效期。异常收货（数量不符、破损等）支持整单撤销并触发线下沟通，撤销后发货方回退至"待重新出库"。

| 单据类型    | 字典        | 是       | 采购入库 / 分销入库 / 借货入库        |
| 关联出库单  | 文本        | 是       | 发货方单号                            |
| 入库仓库    | 字典        | 是       | 经销商仓库列表                        |
| 产品明细    | 结构        | 是       | 产品/批号/序列号/生产日期/有效期/数量 |
| 实收数量    | 数值        | 是       | ≤ 应收数量                            |
| 备注 / 附件 | 文本 / 文件 | 否       | 异常时必填原因                        |

**6.B.5 库存管理**

**模块概述**

库存模块是进销存的数据中枢，涵盖 9 个子功能：仓库维护、实时库存查询、库存汇总、库存调整、移库、借货出入库、库存盘点/月末上传、退换货、产品流向追踪。所有库存变动均产生"库存流水"，支持按批号、序列号、生产日期、有效期、入库来源等多维追溯，为医疗器械 UDI、快消召回、生鲜溯源提供底层能力。

**6.B.5.1 仓库维护**

经销商可维护主仓库、分仓库、医院仓库三类仓库。主仓库有且仅有一个（不可删除），分仓库可创建多个，医院仓库需选择关联医院并自动带出地址。仓库停用前系统检查是否有余额或未完成单据，若有则阻止停用。

|          |          |              |                        |
|----------|----------|--------------|------------------------|
| **字段** | **类型** | **必填**     | **说明**               |
| 仓库名称 | 文本     | 是           | ≤ 50 字符              |
| 仓库类型 | 字典     | 是           | 主仓库/分仓库/医院仓库 |
| 所在医院 | 关联     | 医院仓库必填 | 引用医院主数据         |
| 地址     | 结构     | 是           | 省/市/区/详细          |
| 状态     | 字典     | 是           | 启用/关闭              |

**6.B.5.2 库存查询 / 库存汇总**

库存查询提供明细视图（每一条记录 = 产品 + 批号 + 序列号 + 仓库），支持按分公司、经销商、仓库、部门、产品、批号、序列号、生产日期、有效期、入库来源多维筛选与 Excel 导出；库存汇总提供按产品维度的汇总视图，适合月末对账使用。二者共用同一查询条件区。经销商用户仅可见本经销商库存，一级经销商可下钻查看其二级的分销入库数据（受授权约束）。

**6.B.5.3 库存调整**

分为"增加类"与"扣减类"两大类：

- 增加类：换货入库、厂方换货、库存盘盈、其他入库 → 需商务审批后方可入账；

- 扣减类：过期报损、损坏报损、库存盘亏、其他出库 → 无需审批，直接扣减。

用户填写调整类型、原因、明细（产品/批号/序列号/数量），保存草稿或提交。增加类进入待审批队列，商务审批通过后库存增加；扣减类提交后立即生效。所有调整均写入库存流水与审计日志，作为月末盘点差异分析的依据。

|                |                  |              |                    |
|----------------|------------------|--------------|--------------------|
| **当前状态**   | **允许操作**     | **下一状态** | **触发条件**       |
| 草稿           | 编辑/删除/提交   | 提交 / —     | 发起人             |
| 提交（增加类） | 审批通过/拒绝    | 同意/拒绝    | 商务审批           |
| 提交（扣减类） | （自动通过）     | 完成         | 提交即生效         |
| 同意           | （自动完成）     | 完成         | 审批通过后立即入账 |
| 拒绝           | 基于原单再次发起 | 草稿         | 发起人操作         |

**6.B.5.4 移库**

经销商内部仓库之间的库存转移，总库存不变。用户新建移库单 → 选择源仓库/目标仓库 → 添加产品与数量（校验源仓库可用库存 ≥ 移出数量）→ 保存草稿或提交。提交后源仓库库存立即扣减、目标仓库库存立即增加，不再需要收货确认。移库单不可删除，仅可撤销（撤销自动生成反向移库）。

**6.B.5.5 借货出库 / 借货入库**

用于经销商之间在紧急场景下的临时调拨，货权仍归借出方，借入方接货后需在约定时间内归还或采购结算。借出方新建借货出库单 → 选择借入方 → 添加产品 → 提交。提交后借入方 DMS 自动生成对应"借货入库"待接收单据；借入方未确认收货前借出方可撤单，确认后不可撤。归还时通过反向的借货单据完成。

|              |                       |              |              |
|--------------|-----------------------|--------------|--------------|
| **当前状态** | **允许操作**          | **下一状态** | **触发条件** |
| 草稿         | 编辑/删除/提交        | 待接收       | 借出方提交   |
| 待接收       | 借入方确认/借出方撤销 | 完成/撤销    | 双方操作     |
| 完成         | 归还流程              | （新单）     | 归还发起     |

**6.B.5.6 库存盘点 / 月末库存上传**

经销商每月最后 3 个工作日内上传月末库存快照，含实盘库存 + 寄售库存。系统提供模板下载 → 填写 → 上传 → 系统校验 → 生成差异报告；允许在窗口期内重复上传，覆盖上一次结果。差异报告展示"系统账面 vs 上报实盘"的差异行，用于经销商与厂商对账、指导库存调整。次月上传视为迟报，影响 KPI 中的"上报及时率"指标。

上传前系统提示当前月份未完成的收货、销售、调整、移库、借货单据，建议全部完成后再盘点；仍未完成的单据不影响上传，但会记入差异行。

**6.B.5.7 退换货**

用于向厂商或上级经销商退回不良品、过期品、错发品。发起前必须持有由厂商销售或商务下发的"退货授权编号"（RMA），授权含额度与 2 年时限，一次授权可支持多张退换货单直至额度用尽。经销商新建退换货单 → 引用授权编号 → 选择退货对象与单据类型（退货/换货）→ 添加产品明细（批号/序列号必填）→ 说明退货原因 → 上传附件 → 提交。

审批通过后系统生成退货单号（作为寄回包裹的凭证），经销商按流程随货寄回；实际货物到达厂商仓后由厂商侧确认收货并触发库存扣减（退换货不冻结库存，按审批结果实际扣减）。

|              |                 |              |              |
|--------------|-----------------|--------------|--------------|
| **当前状态** | **允许操作**    | **下一状态** | **触发条件** |
| 草稿         | 编辑/删除/提交  | 提交         | 发起人       |
| 提交         | 审批通过/拒绝   | 同意/拒绝    | 商务审批     |
| 同意         | 打印退货单/寄回 | 寄回中       | 发起人操作   |
| 寄回中       | 厂商收货        | 完成         | 厂商确认     |

**6.B.5.8 产品流向追踪**

系统对每一次入库、出库、移库、销售出库均记录批号、序列号、生产日期、有效期、来源单据，构成完整的产品流向链路。用户可在流向追踪页按产品编码 + 批号/序列号定位单条流向记录，页面以时间轴形式展示"厂商发货 → 经销商入库 → 移库/借货 → 销售给终端"的完整轨迹。该模块为医疗器械 UDI 追溯、快消召回、生鲜溯源提供数据支持，导出结果可直接对接监管平台。

**6.B.6 销售管理**

**模块概述**

覆盖两种"出库到下游"的场景：销售出库单（对终端客户/医院销售，含手术报台/植入信息）与分销出库（一级经销商向二级/三级经销商分销）。医疗器械行业销售出库单同时承担"报台"责任，以手术为单位、不按产品线拆分、不含价格；快消行业销售出库单可简化为按订单/门店出库。

**6.B.6.1 销售出库单（含报台）**

用户新建销售出库单 → 选择销售终端（必填）→ 填写手术/销售信息（医疗器械行业为植入单号、医生、专业、科室、手术类型、手术日期、疾病；病人姓名/性别/年龄；快消行业为客户单号、门店编号、销售日期）→ 添加产品明细（仓库、产品、序列号/批号、数量）→ 上传附件（必填，为植入单/客户签收单扫描件）→ 保存草稿 或 提交。提交后库存实时扣减、销售流水生成、Sales-Out 数据回传厂商用于 IMS 与返利结算。

单据支持"冲红"（红字冲销）：对已提交且被发现有误的单据，创建反向单据抵销原单，原单和红冲单均保留。经销商必须在次月第 5 个工作日前完成本月所有销售数据上报，超期视为迟报，影响 KPI。

|                     |          |          |                             |
|---------------------|----------|----------|-----------------------------|
| **字段**            | **类型** | **必填** | **说明**                    |
| 销售出库单号        | 文本     | 是       | 自动生成                    |
| 销售终端            | 关联     | 是       | 引用授权列表                |
| 植入单号 / 客户单号 | 文本     | 是       | 业务单号                    |
| 手术信息 / 销售信息 | 结构     | 是       | 医生/科室/日期 或 门店/日期 |
| 产品明细            | 结构     | 是       | 仓库/产品/序列号/批号/数量  |
| 附件                | 文件     | 是       | ≤ 20MB/文件                 |
| 备注                | 文本     | 否       | ≤ 500 字符                  |

**6.B.6.2 分销出库**

一级经销商向下游二级/三级经销商分销时创建。以下游订单为基准，一张分销出库单对应一个下游订单（不允许一对多）。用户在下游订单列表中选择"生成分销出库" → 系统自动带出订单明细 → 用户填写批号/序列号/快递信息 → 提交。提交后下游经销商 DMS 自动生成对应"采购入库/分销入库"待接收单据，形成 A2A（Agent-to-Agent）链路。厂商侧可基于分销出库数据判断层级动销结构。

|              |                   |              |              |
|--------------|-------------------|--------------|--------------|
| **当前状态** | **允许操作**      | **下一状态** | **触发条件** |
| 草稿         | 编辑/删除/提交    | 待接收       | 一级操作     |
| 待接收       | 下游确认/一级撤销 | 完成/撤销    | 双方操作     |

**6.B.7 发票管理**

包含两类：厂商发给经销商的采购发票（跟随订单流程上传/确认）与经销商发给终端的销售发票（跟随销售出库单上传）。发票信息包含发票号、开票日期、金额、税额、图片（扫描件或电子发票 PDF）；发票与订单/销售单据一对一或一对多关联。发票支持批量上传（Excel 模板 + 图片压缩包）；部分产品线可配置"发票+植入单强制绑定"，否则销售出库单无法提交。所有发票扫描件保存 5 年以上供审计使用。

**6.B.8 门户 / 首页**

登录后的第一屏，提供跨模块信息聚合。经销商侧与厂商内部用户侧共用同一首页框架，但组件按角色差异化展示。

- 待办任务：按数量气泡提示各模块待处理事项（待审批订单、待收货、待上传发票、待盘点等）；

- 公告通知：厂商发布的政策、活动、系统升级公告，支持置顶与阅读状态；

- 日历：本月的订单截止日、库存上报截止日、销售上报截止日；

- 指标看板：本月达成率、同比增长、Sales-In vs Sales-Out 对比、IMS 数据；

- 文件下载：厂商下发的价格表、政策文件、操作手册、模板文件；

- 快捷菜单：按角色可配置的常用功能入口；

- 服务热线：厂商客服联系方式与工单入口。

**6.B.9 账户与安全**

账号规范：厂商内部用户账号为工号；经销商用户账号为"经销商编号 + \_序号"（如 \`D00123_01\`），其中 \`\_01\` 为该经销商主账号（admin），可管理下属子账号。首次登录强制修改初始密码（默认 \`Sh123456\`）。密码策略：≥6 位、字母+数字组合、180 天到期、连错 9 次锁定。锁定后由管理员在"用户管理"中执行"Unlock User"解锁。经销商主账号可为本经销商新增子账号，重置密码后系统通过邮件发送新密码给指定邮箱。

**6.C 报表与经销商画像 (RS & DP)**

**6.C.1 Reporting Service（固定报表）**

对合同、订单、库存、销售、授权、借货、发票、返利、折扣、退货等主题提供预定义报表。每张报表遵循统一交互模式：顶部条件区 → 中部数据网格 → 底部汇总行；支持 Excel 导出、图表切换（柱/折/饼）、按维度下钻。KPI 类报表额外提供环比、同比、达成率的可视化对比。

- 合同报表：合同信息汇总/明细、合同状态、签章信息、操作日志、待处理单据；

- 订单报表：订单量、拒单率、审批时长、订单结构；

- 库存报表：库存周转、呆滞库存、临期库存、盘点差异；

- 销售报表：Sales-Out 明细、终端分布、产品排名、季节性；

- 授权/借货报表：授权矩阵、借货余额、超期未还；

- 发票/返利/折扣：财务对账支持；

- 系统访问记录、申请流程报表：合规与运维。

**6.C.2 返利与指标计算**

系统根据合同中的指标（年度采购额、Sales-Out 目标、终端覆盖数）与实际发生数据（订单、销售、库存）自动计算达成率、返利金额、折扣金额。计算结果按月生成并支持"预演"（在结算前经销商可查看预计返利）。返利规则支持分段（如达成 80% 返 X%、达成 100% 返 Y%）、封顶、扣减（因串货、退货、迟报扣减返利）等复杂场景。计算过程与最终数字对经销商透明，减少对账争议。

**6.C.3 Dealer Profile（经销商 360 画像）**

对每一家经销商提供一站式全景视图。左侧信息导航、右侧模块化卡片，按需展开：

- 基本信息：法人、注册地址、经营范围、股东结构；

- 审计信息 / QA 信息：历次审计结果、GSP/GMP 证书、合规风险等级；

- 达成明细：本年度累计达成、月度趋势、TOP 产品；

- KPI 明细：上报及时率、订单通过率、退货率、投诉率；

- 返利明细：本年度累计返利、按季度拆分；

- 指标信息：本年度指标、剩余任务、达成路径；

- 合同信息：当前生效合同、历史合同、即将到期提示；

- 培训记录 / 问卷调研：厂商组织的培训与调研的参与记录；

- 进销存信息：库存快照、Sales-In、Sales-Out、周转率；

- 关联经销商：上下级关系、共同股东、地理临近；

- 附件：证照、合同、盖章文件、往来邮件。

**6.C.4 Dashboard（综合看板）**

面向管理层的多维度综合看板，默认展示全国维度，可下钻到大区/省份/经销商。核心组件：本月 Sales-In vs Sales-Out 对比、达成 TOP10 / BOTTOM10 经销商、地图热力（销售、库存、经销商密度）、临期库存预警、串货预警、指标达成雷达图。看板支持时间维度切换（本日/本周/本月/本季/本年）与数据钻取，点击图表元素可跳转至对应报表或 Dealer Profile。

**6.D 系统管理（通用能力）**

**6.D.1 权限模型（RBAC 四层）**

平台采用"组织 → 资源 → 策略 → 角色"的四层 RBAC 模型：

- 组织架构（Organizational Units）：按厂商实际组织树维护节点，用户挂在节点上以确定数据可见范围；

- 资源（Resources）：菜单、按钮、API、报表等被授权对象，每个资源可定义读/写/删除/审批等操作；

- 策略（Strategy）：一组资源 + 操作的组合，可命名并复用；

- 角色（Role）：一个或多个策略的组合，最终授予用户。

数据权限维度独立于角色：厂商内部用户基于组织架构做数据隔离；经销商用户按经销商 ID 强隔离。所有主数据、单据的查询接口需按数据权限自动过滤。

|          |                             |                                       |
|----------|-----------------------------|---------------------------------------|
| **层级** | **主要字段**                | **操作**                              |
| 组织     | 节点名、上级、层级          | Add / Add Child / Edit / Add Users    |
| 资源     | 资源名、支持操作、菜单归属  | Edit → Support Operation → Save       |
| 策略     | 策略名、描述、资源-权限组合 | Add / Edit → Update Permission → Save |
| 角色     | 角色名、描述、策略列表      | Add / Edit → Add Strategy → Save      |

**6.D.2 用户管理**

用户字段包含账号、姓名、类型（User/Dealer）、状态、电话、邮件、组织架构、角色、自定义扩展字段（如销售条线：设备/非设备）。厂商内部用户由管理员创建，经销商用户由厂商管理员在"经销商信息及授权"页新增（也可由经销商主账号自建子账号）。操作：Add / Edit → Add Attributes（组织/角色）→ Save；Unlock User（解锁）；Reset Password（重置并邮件发送）。

**6.D.3 流程设定**

用于配置各审批流（合同申请、订单、库存调整、退换货等）的节点、审批人、条件分支。按流程类型模糊查询 → 新建流程 → 定义节点 → 配置字段可见性 → 保存。流程一经绑定即立即生效，正在流转的单据仍按原流程走完全流程，新单据走新流程。

**6.D.4 字典表与菜单管理**

字典表：全局枚举值维护（订单类型、状态、单据类型、终端类型等）。菜单管理（SiteMap）：一/二级菜单结构维护，与资源模块联动。二者主要由开发协助管理员配置，日常使用频率低。

**6.D.5 日志与缓存监视**

审计日志（Auditing Log）：分类查询主数据变更、后台管理动作，可下钻查看某条记录的全部变更历史（前值、后值、操作人、时间）。缓存监视：显示各主数据缓存状态（已更新/待更新），支持手动 Run 刷新，并每晚定时全量刷新一次。

**6.D.6 信息管理（门户内容）**

- 公告通知维护：新增公告 → 选择目标经销商范围 → 保存草稿或发布；

- 常见问题维护：FAQ 增删改查；

- 下载文件管理：上传/下架政策文件、模板、手册；

- 经销商用户维护：厂商侧管理经销商账号的入口（对应 6.D.2 的角色视角）。

**6.E 外部系统接口**

DMS 通过标准接口与厂商 IT 生态其他系统集成，采用 REST API + 消息队列组合，保证实时性与最终一致性。

|----|----|----|
| **系统** | **方向** | **主要接口** |
| ERP | ERP → DMS | 产品主数据、产品价格、发票信息、发货信息、信用额度、经销商同步 |
| ERP | DMS → ERP | 订单推送、合同经销商同步、销售数据回写、发票登记 |
| WMS | 双向 | 仓库出入库指令与回执 |

**7. 核心业务流程**

本章以端到端流程视角描述跨模块协同，供实施与培训参考。

**7.1 经销商合同全生命周期**

主数据准备 → 合同申请（新建/变更/续约/终止/批量）→ 工作流审批（多节点，可邮件审批）→ 生成协议 PDF（按模板）或上传纸质扫描件 → 双方电子签章（或线下盖章 + 扫描回传）→ 法务归档 → ERP 同步 → 授权/指标/价格/返利/折扣主数据生效。

**7.2 主数据与授权下发**

ERP 产品主数据 / 价格 / HR 人员 → 接口同步 → DMS 主数据 → 合同产生的授权（终端/区域/产品线）→ 授权控制订单/销售/退换货/移库/借货有效期。

**7.3 订单到收货全流程**

经销商网上下单（按授权）→ 提交 → 客服审批（可拆单）→ ERP 下单/供应链确认 → ERP 发货 → DMS 自动生成收货单 → 经销商整单收货（选仓库入库）→ 发票上传/确认 → 订单关闭。

**7.4 库存日常管理**

收货入库 → 移库/借货出入库 → 库存调整（增加类需审批 / 扣减类免审）→ 实时库存查询 → 每月末最后 3 个工作日上传月末库存（差异比较）→ 差异调整。

**7.5 销售上报（含报台）**

经销商实际销售/植入 → 销售出库单录入（销量+销售/植入信息，序列号查重，不含价格）→ 单据可撤销/冲红 → 上传发票扫描件（部分产品线强制）→ 次月第 5 个工作日前上报。

**7.6 分销出库（一级→下级）**

一级经销商基于下级订单生成分销出库单（一单对应一订单）→ 下级 DMS 自动生成采购入库 → 下级确认收货 → 上级库存扣减、下级库存增加。

**7.7 借货流程**

借出方生成借货出库单 → 借入方 DMS 自动生成借货入库单 → 借入方确认收货前借出方可撤单 → 借入方确认后完成库存转移。

**7.8 退换货流程**

销售/渠道人员在 DMS 申请经销商退货授权（额度 + 2 年时限）→ 经销商发起退换货申请（引用授权编号）→ 商务审批 → 打印退货单号随货寄回 → 厂商收货后库存扣减。

**7.9 产品流向追踪**

每次入库/出库/移库/销售出库均记录批号、序列号、生产日期、有效期、来源 → 形成产品流向报表 → 支撑 UDI 追溯 / 快消召回 / 生鲜溯源。

**7.10 经销商上线（首用）**

导入期初库存 → 校验仓库信息 → 修改姓名/邮箱/联系方式 → 首登强制改密 → 连续两个月数据校验期（票据扫描件由第三方核验）→ 转入正常运营。

**8. 非功能性需求**

**8.1 性能**

- 页面加载 ≤ 2s（首屏），列表查询 ≤ 1.5s（10 万级数据分页）；

- 订单提交、审批、收货等关键操作响应 ≤ 800ms；

- 并发：支持 500 并发在线用户、50 TPS 订单提交、1000 QPS 报表查询；

- 批量导入：单次 ≤ 10000 行，10 万行以内需分批。

**8.2 可用性**

- 系统年可用率 ≥ 99.5%（工作日 8:00–22:00 内可用率 ≥ 99.9%）；

- 计划内停机窗口：每月最后一个周日 22:00–次日 06:00；

- 灾备：核心数据每日全量备份 + 每小时增量备份，RTO ≤ 4h，RPO ≤ 1h。

**8.3 安全**

- 传输层：全站 HTTPS，TLS 1.2 以上；对外接口 IP 白名单；

- 存储层：敏感字段（身份证、银行卡、密码）加密存储；密码使用 bcrypt/argon2；

- 访问层：RBAC + 数据权限，关键操作二次认证（短信验证码/UKey）；

- 应用层：防护 SQL 注入、XSS、CSRF、路径穿越、命令注入、日志伪造；

- 审计：所有登录、主数据变更、单据审批、签章操作全量落审计日志，保留 3 年。

**8.4 合规**

- 医疗器械行业：满足 UDI 追溯要求，支持批号 + 序列号双维度追踪；

- 快消 / 食药：满足溯源、召回、临期管理监管要求；

- 数据保护：满足《个人信息保护法》《数据安全法》，敏感信息脱敏展示；

- 电子签章：满足《电子签名法》，采用有资质的第三方 CA。

**8.5 部署与兼容**

- 部署形态：私有云（Azure/AWS/阿里云）或 IDC 内部部署；

- 浏览器：Chrome / Edge 最新两个版本，兼容 IE 11 只读；

- 前端：响应式，最小分辨率 1366×768；不做原生 App，可考虑 H5 移动端；

- 语言：中文简体（默认）、英文；预留多语言扩展。

**9. 遗留问题与假设**

本 PRD 基于对原始材料的整合形成，以下事项仍需在评审阶段与业务方确认：

- \[待确认\] 快消行业销售出库单是否强制上传附件？（医疗器械强制，快消可能不需要）

- \[待确认\] 二级经销商是否统一使用本平台？部分厂商仅要求一级经销商使用。

- \[待确认\] 返利规则的最大分段数与最大扣减项数量；

- \[待确认\] UDI 上报是否直接对接监管平台，还是仅在 DMS 内留存；

- \[待确认\] 是否需要移动端 H5（部分场景如仓管扫码收货可能需要）；

- \[假设\] 电子签章采用第三方 CA 服务，不自建 CA；

- \[假设\] ERP、HR、WMS 已具备开放 API，DMS 侧仅做集成，不改造上游系统；

- \[假设\] 数据留存 5 年（发票、合同）/3 年（审计日志）为默认要求。
