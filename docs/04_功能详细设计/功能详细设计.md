## FDD-1 登录与账户

> V1 不做 SSO：OIDC / employee_id 换 user 等集成方案推迟到 V2；V1 登录仅支持「账号密码」与「微信扫码」两种方式。

### 1.1 登录
**输入**：`username`、`password`、`remember_me`、`captcha`（连续错误 3 次后启用）
**校验**：
1. 用户名/密码空 → `40001 参数缺失`；
2. 图形验证码错误 → `40002`；
3. 账号不存在 或 密码错误 → 统一返回 `40101 账号或密码错误`（不区分）；
4. 用户 `status=locked` → `40102 账号已锁定`；
5. 用户 `tenant.status!=active` → `40103 租户不可用`；
6. 成功后：
   - 生成 JWT（`access` 1h + `refresh` 7d）；`remember_me=true` 时 refresh 延长至 7d 存 cookie；
   - 写审计：`AUDIT.action=LOGIN, ip, ua, tenant_id, user_id`；
   - 若 `must_change_password=true` → 302 到改密页；
   - 触发 `login_fail_count = 0`。

**状态字段**：`user.status ∈ {active, locked, disabled}`；`login_fail_count`, `locked_until`。
**输出**：`{access_token, refresh_token, user{id, name, roles[], tenant_id, permissions[], must_change_password}}`
**异常**：
- 密码错误：`login_fail_count += 1`；累计 ≥ 9 → `status=locked, locked_until = now+30min`；
- 锁定期内继续尝试 → `40102`；到期后 `unlock` 定时任务恢复；
- 手动解锁：管理员在用户管理页触发（写审计）。

### 1.2 忘记密码
**输入**：邮箱 + 图形验证码
**校验**：
- 邮箱不存在 → 依然返回 200（防枚举），但不发邮件；
- 15 分钟内同邮箱最多 3 次；超频 → `42901`。
**输出**：
- 发送邮件（HTML 模板），链接含 15 分钟一次性 `reset_token`；
- 用户点击链接 → 校验 token → 新密码表单；
- 提交后：`password_hash` 更新（bcrypt），`must_change_password=false`，`login_fail_count=0`，全部旧 session 失效；
- Toast 后跳回登录页。

### 1.3 首次登录强制改密
- 登录后 middleware 检测 `must_change_password=true` → 强制跳 `/force-change-password`；
- 校验策略：≥6 位、含字母+数字、不等于旧密码；
- 提交成功后 must_change_password=false，password_updated_at 更新，重新登录。

### 1.4 SSO（Mock）
- V1 已删除；OIDC / `X-SSO-Employee-Id` Mock 桩相关内容不再实现，端点从 FDD-14 外部接口中一并移除。

### 1.5 微信扫码登录
**目标**：H5 与 PC 端登录页均提供「微信扫码登录」入口，首次扫码需绑定一次 DMS 账号，后续免密。

**流程（H5 端）**：
1. 用户点击「微信扫码登录」→ 前端调用 `GET /api/wechat/qr?scene=login` → 后端调用微信开放平台生成带 scene 的二维码，返回 `qr_url + ticket`；
2. 前端展示二维码，同时开启轮询 `GET /api/wechat/qr/status?ticket=xxx`（或使用 WebSocket）；
3. 用户在微信中扫码授权 → 微信回调 `POST /api/wechat/callback`（携带 code）→ 后端换取 `openid + unionid`；
4. 分支：
   - `openid` 已存在于 `users.wechat_openid` → 校验用户 `status=active` 且租户可用 → 直接颁发 JWT（同 1.1 成功流程，写审计 `action=LOGIN_WECHAT`）；
   - `openid` 未绑定 → 返回 `binding_required=true` + 一次性 `bind_token`（10 分钟）→ 前端跳「绑定 DMS 账号」页；
5. 绑定页：用户输入 DMS `username + password`（复用 1.1 校验规则）→ `POST /api/wechat/bind` body=`{bind_token, username, password}` → 校验通过后写入 `users.wechat_openid, wechat_unionid, wechat_bound_at` → 同步颁发 JWT；
6. 后续再次扫码即命中步骤 4 的直登分支，实现免密。

**PC 端**：登录页在账号密码表单旁增加「微信扫码」Tab，二维码流程与 H5 一致；扫码后 PC 端页面轮询到状态 `SCANNED_CONFIRMED` 后自动跳转工作台。

**安全**：
- `bind_token` 一次性，绑定成功或过期即失效；
- 一个 DMS 账号仅能绑定一个 `wechat_openid`（唯一索引），重复绑定报 `40901 微信已被其他账号绑定`；
- 解绑：用户在「我的-账号安全」中解绑，写审计。

**异常**：
- 微信回调 code 无效 → `40201 微信授权失败`；
- 用户拒绝授权 → 前端提示重新扫码；
- 绑定阶段账号密码错误 → 复用 `40101`。

## FDD-2 工作台首页
**输入**：无（当前 session 用户 + tenant + role）
**API**：`GET /api/home/dashboard`
**服务端聚合**（并行）：
- 本月订单量 = orders.count(tenant, submit_time in month)
- 销售额(万元) = sum(order.amount_incl_tax) / 10000
- 待审批订单 = orders.count(status='待审批' AND assignee=me)
- 库存预警 = inventory.count(avail_qty <= safety_qty)
- Sales-In vs Sales-Out 近 6 月（月粒度）
- 订单状态分布 环形数据
- 待办 = 待审批 + 待收货 + 待签章 + 待盘点 + 资质到期 前 10 条
- 公告 top 5（含置顶）
- 消息未读数

> **消息通道（V1）**：待办气泡与业务通知统一走「站内消息 + 企业微信 / 飞书 Webhook」两条渠道；V1 已删除邮件与短信通道，相关模板与 `/mail/send`、`/sms/send` Mock 端点一并下线。Webhook 目标地址在 `system_settings.notify.webhook.*` 中按租户配置。

**输出**：JSON 聚合结果（含每张卡片的环比"vs 上月"）
**异常**：某个子查询失败 → 部分数据缺失但整体不 500，前端显示 `─ 加载失败`

## FDD-3 合同管理

### 3.1 合同申请
**输入**：`type ∈ {NEW, MODIFY, RENEW, TERMINATE, BATCH_EXTEND, BATCH_UPDATE}`
**必填字段**（按类型动态）：
- NEW：contract_category, dealer_info, authorization{terminals, product_lines, regions, valid_from, valid_to}, indicators, attachments[资质]
- MODIFY：ref_contract_id, diff_fields{}
- RENEW：ref_contract_id, new_valid_to
- TERMINATE：ref_contract_id, reason, effective_date
- BATCH_EXTEND：contract_ids[], new_valid_to
- BATCH_UPDATE：contract_ids[], patch{}

**校验**：
1. valid_from ≥ today；valid_to > valid_from；
2. 附件必填项按 contract_category 从 `dict.contract_attachment_matrix` 校验；
3. dealer 唯一性（同类型下不允许同一 dealer 存在活跃申请）；
4. TERMINATE 需 ref_contract.status='生效'；
5. 每次 modify 生成变更前后 JSON 对照存 `contract_diff`。

**状态流转**（表 `contract_applications`）：
```
草稿 → 待审批 → 审批中 → 待签章 → 生效
                ↓          ↓
              拒绝(可再发) 拒绝
生效 → (子流程 MODIFY/RENEW/TERMINATE) → 新申请
```
**输出**：申请单号 + 审批任务分配
**异常**：
- 附件 ClamAV Mock 扫毒失败 → `40006 附件病毒风险`；
- 审批过程中 dealer 数据被删 → 阻断继续，回退发起人；

### 3.2 审批
- 审批任务表 `approval_tasks`：`ref_type, ref_id, node_id, assignee, deadline, status`；
- 审批 API `POST /api/approvals/{task_id}/action` body `{action, comment, next_assignee?}`；
- action ∈ `PASS / REJECT / TRANSFER / ADD_SIGN`；REJECT 必填 comment(≥5 字)；
- 24h/48h/72h 超时定时器（Quartz-like，Redis TTL）发送提醒/加急/升级消息；
- 邮件审批 Token：`GET /open/approval?token=xxx` 免登进入审批页；Token 单次使用+24h 过期。

### 3.3 电子签章
- 审批通过后 job `contract.generate_pdf` 5min 内执行，模板引擎（Freemarker）→ 落入 MinIO；
- 经销商侧签章：`POST /api/contracts/{id}/sign/dealer` 校验 sms_code，回写 `dealer_sign_time, ca_serial_no`；
- 厂商侧签章：`POST /api/contracts/{id}/sign/vendor`；
- 双签完成 → 触发 event `contract.effective` → `erp.sync (Mock)` + `legal.archive`；
- 失败重试队列 3 次，指数退避。

## FDD-4 授权管理

### 4.1 合同授权（自动生成）
合同生效后事件监听 → 拆解 `authorizations`：dealer × product_line × terminal × auth_type × valid_from/to。

### 4.2 临时授权
**输入**：dealer_id, auth_type, targets[], valid_from/to, reason, attachments
**校验**：valid_to > valid_from；不能与既有生效授权重复；
**状态**：`申请中 → (审批) → 生效 / 拒绝 → 到期失效`；
**授权校验函数**（订单/销售/退换货/移库/借货提交时调用）：
```
authorized(tenant, dealer, product_id, terminal_id, auth_type, at_time)
  = EXISTS(authorizations WHERE dealer AND auth_type
     AND product_line matches AND terminal matches
     AND status='生效' AND at_time BETWEEN valid_from AND valid_to)
```
不通过时返回具体失败行列表（`product_id + terminal_id + auth_type`）。

### 4.3 终止授权
合同 TERMINATE 生效 → 相关 authorization.status='失效' 立即封锁；无窗口期（D-07）。

## FDD-5 订单管理

### 5.1 新建订单
**输入**：`order_type ∈ {PURCHASE, SHORTAGE, CUSTOM}`, `dealer_id`, `lines[]`, `ship_address`, `remark`
**line**：`product_id, qty, unit_price(自动带出), sub_total`
**服务端流程**（POST /api/orders）：
1. 校验授权（FDD-4.2 authorized）→ 失败返回失败行；
2. 单价 = product.current_price（不再落"下单价"，D-06）；
3. **触发促销引擎**（FDD-10.4）→ 返回 `promotions_hit[]` 和 `discount_amount, gifts[]`；
4. 保存订单主 + 明细 + 促销命中；
5. 生成单号 `ORD-YYYYMMDD-NNNNNN`（序列号由 PG sequence + 日期）；
6. 状态=`草稿` 或 `已提交待审批`（按 action）；
7. 审批任务分配（客服角色 + 组织归属 dealer 负责人）。

### 5.2 审批
- Action：`PASS / REJECT / SPLIT / REBACK`（退回修改）；
- SPLIT：把明细拆成 N 张子单（`parent_order_id` 建立追溯链）；
- PASS 后 → 事件 `order.approved` → `erp.push (Mock)` → 状态推进；
- 状态机（表 `orders.status`）：
```
草稿
 → 已提交待审批
   → 审批通过 → 已进入 ERP 系统审批
                 → ERP 已反馈 → ERP 已发货
                                 → 已确认（经销商收货）
                                 → 已完成（发票上传）
                                 → 已关闭
   → 审批拒绝
   → 等待经销商修改（REBACK）→ 已提交待审批
 → 已撤销
```
- 撤销：仅 `草稿`、`已提交待审批 且未开始审批` 可撤。

### 5.3 批量导入
- Excel 模板 8 列（dealer, product_code, qty, ship_prov, ship_city, ship_addr, contact, phone）；
- 单次 ≤10000 行；
- 服务端逐行校验（授权/库存/字段）→ 输出结果 Excel（新增"失败原因"列）；
- 前端展示失败行红标 → 用户修正 → 二次提交。

### 5.4 异步导出
- POST /api/orders/export → 返回 `job_id`；
- 后台 Job 生成 Excel 落 MinIO → 站内消息推送下载链接；
- 单次 ≤10 万行，超出报错。

## FDD-6 收货管理

### 6.1 待收货生成
- 事件 `shipment.created` (来自 ERP Mock / 分销出库 / 借货出库) → 生成 `receipts` 单据 `status=待接收`；

### 6.2 确认收货
**输入**：receipt_id, lines[{ product, batch_no, serial_no, qty, prod_date, exp_date, warehouse_id }]
**校验**：
1. 每行 qty ≤ 应收 qty；
2. 医疗器械行业（tenant.industry='medical'）：`serial_no` 必填；
3. serial_no 全局唯一（tenant 内）→ 重复报错；
4. warehouse.status='enabled'。

**副作用**：
- 更新 `inventory` 表（+qty）；
- 写 `inventory_transactions`（in_source='采购入库/分销/借货'）；
- 生成 UDI 轨迹记录 `product_trace`。

### 6.3 异常收货
- 整单撤销 → `receipts.status='已取消'`；
- 发货方对应 `shipments.status='待重新出库'`；
- 记原因附件。

## FDD-7 库存管理

### 7.1 仓库维护
- warehouse_type ∈ {main, sub, hospital}；
- main 每 dealer 唯一（DB 唯一索引）；
- hospital 需关联 hospital_id 且带出地址；
- 停用前检查 inventory.qty=0 AND 无未完成单据；否则拒绝。

### 7.2 库存查询/汇总
- 明细：inventory JOIN product JOIN warehouse；索引 (`tenant_id`, `dealer_id`, `product_id`, `batch_no`, `serial_no`)；
- 汇总：GROUP BY product_id, dealer_id；
- 数据权限过滤：经销商仅本人；一级下钻二级需授权。

### 7.3 库存调整
- 类型 20+ 种，分增/扣两类；
- 表 `inventory_adjustments`；
- 增加类：提交 → 待审批（商务）→ 通过后写 inventory + transactions；
- 扣减类：提交 → 立即写库；
- 状态：`草稿 / 待审批 / 已通过 / 已拒绝 / 已生效`。

### 7.4 移库
- 表 `stock_moves`；
- 校验源仓库可用库存 ≥ 移出；
- 提交即生效（原子事务：源 -qty，目标 +qty，写 transactions ×2）；
- 撤销 → 生成反向移库。

### 7.5 借货
- 表 `loans`；
- 借出方创建 → 借入方待接收 → 确认收货 → 双方库存变动；
- 借入方未确认前借出方可撤单；
- 归还 → 反向借货单。

### 7.6 盘点上传
- 表 `stocktakes`（月粒度）与 `stocktake_lines`；
- 窗口 = 当月最后 3 个工作日（工作日日历表 `working_days`）；
- 支持窗口内重复覆盖；
- 差异报告 = 系统账面 vs 上传实盘；
- 迟报仅打标 `is_late=true`，影响 KPI。

### 7.7 退换货 & RMA 授权
- `rma_authorizations`：dealer, product_line, quota_amount, quota_used, valid_from/to(≤2y)；
- `rma_orders`：ref_rma_auth_id, type='退货/换货', lines, reason, attachments；
- 状态：`草稿 → 提交 → 商务审批 → 打印退货单 → 寄回中 → 厂商收货 → 完成`；
- 授权额度实时扣减（rma_orders.amount 提交时占用）；到期或额度用尽后不允许发起新单。

### 7.8 产品流向追踪
- 表 `product_trace`（每一次入/出/移/销都追加）：`event_type, ref_doc, product, batch, serial, qty, from_warehouse, to_warehouse, terminal, at_time`；
- 查询接口按 `product_id + batch_no + serial_no` 返回时间轴；
- 导出 Excel/PDF；UDI 上报（Mock）按监管平台协议。

## FDD-8 销售管理

### 8.1 销售出库单
**输入**：dealer_id, terminal_id, surgery_info{}, lines[], attachments[]
**校验**：
- 授权（sales_to_hospital）通过；
- serial_no 未曾报过（防重）；
- 医疗器械：surgery_info 与 attachments 均必填；
- lines.qty ≤ 库存可用；

**副作用**：
- 立即扣库存；
- 写 `product_trace event=sale`；

## FDD-10 促销管理

### 10.1 促销规则实体
表 `promotions`：`id, tenant_id, code(PROMO-...), name, type ∈ {MOQ, FULL_REDUCTION}, priority(1-100), valid_from, valid_to, dealers[], product_lines[], products[], status, description`
- 子表 `promotion_rules`（多态 JSONB）存类型规则详情。

> **V1 范围说明**：`type` 枚举 V1 仅启用 `MOQ` 与 `FULL_REDUCTION`；「满赠 GIFT」与「组合销售 BUNDLE」已从 V1 移除并推迟到 V2，但数据表 `type` 枚举字段的取值仍**保留 `GIFT` 与 `BUNDLE`** 以便未来平滑扩展，V1 服务端在创建/更新促销时对这两个值直接拒绝（返回 `40010 促销类型 V1 未启用`）。

### 10.2 规则详情结构
- **MOQ** (起订量)：`{ product_id, min_qty, mode: 'BLOCK'/'WARN' }`
- **FULL_REDUCTION** (满减)：`{ tiers:[{amount>=X, reduce_type:'FIXED'|'PERCENT', reduce_value}], scope:'ORDER'|'PRODUCT', exclusive:true|false }`

### 10.3 促销审批流
- 独立流程类型 `PROMOTION`，节点默认 [销售经理 → 商务 → 财务]；
- 审批通过后 `status='生效'`；变更/暂停/停用同样走审批。

### 10.4 促销引擎（下单时调用）
输入：dealer_id, lines[]
步骤：
1. 查询 `promotions WHERE status='生效' AND now BETWEEN valid_from AND valid_to AND dealer IN OR ALL AND product IN OR ALL`；
2. 按 `priority DESC, created_at ASC` 排序；
3. 逐条计算命中：
   - MOQ：line.qty < min_qty → 若 BLOCK 返回错误，若 WARN 加 warning；
   - FULL_REDUCTION：分订单/产品维度累加 amount → 匹配最高档；
4. 处理互斥/叠加：exclusive=true 时同类型只保留最高优先级；
5. 返回 `{ hits:[], discount_total, warnings:[] }`；
6. 落库 `order_promotion_hits`（订单 × 促销 × 优惠金额），审计。

> **V1 已移除的分支**：`GIFT`（生成赠品行）与 `BUNDLE`（套餐价替换总价）两个处理分支在 V1 促销引擎中不实现，即便历史脏数据中存在此类型也会被步骤 1 过滤（因为 status/type 校验会拒绝生效）。返回体中相应 `gifts[]` 字段已从 V1 输出结构中删除。

### 10.5 促销预览面板 API
`POST /api/promotions/preview` body=草稿订单 → 返回同上；前端右侧滑出。

## FDD-12 返利计算

### 12.1 触发时机
- 每日凌晨定时 job：重算本月至今返利预演；
- 结算日（月末）：正式生成返利单，写 `rebate_settlements`。

### 12.2 分段公式（3–5 档预置）
输入：合同指标 target_amount, 实际 actual_amount, 分段规则
```
achievement_rate = actual_amount / target_amount
select tier from tiers where achievement_rate >= tier.min_rate order by min_rate desc limit 1
rebate = actual_amount * tier.rate - deductions
```
- deductions（默认扣减项模板）：串货金额、退货金额、迟报罚金；
- 结果字段：`{ dealer_id, period, achievement_rate, tier_hit, gross_rebate, deductions[], net_rebate }`

### 12.3 预演 vs 正式
- 表 `rebate_previews`（每日快照）
- 表 `rebate_settlements`（月末锁定）
- 经销商画像"返利明细"读取 previews；财务对账读 settlements。

## 附录 · V3.0 新增功能设计

### F.1 采购订单模块（v3.0 新增）

#### 输入
- 采购类型：`NORMAL / URGENT / RETURN`
- 供应商：必填（从 dealers 表选择，V1 复用）
- 入库仓库：必填
- 期望到货日期
- 采购备注
- 明细行：产品ID / 数量 / 单价 / 税率

#### 校验规则
1. `code` 唯一（自动生成 `PO-{timestamp}`）
2. 明细 `qty > 0` `unit_price >= 0`
3. `tenant_id` 从 JWT 自动填充
4. 编辑时**仅 DRAFT 状态可改**，其他状态改动返回 40009 状态码

#### 状态流转（完整状态机）

```
        创建             提交              审批通过           收货入库
DRAFT ─────► DRAFT ────► SUBMITTED ─────► APPROVED ─────► RECEIVING ─────► COMPLETED
                           │                  │
                           │驳回              │取消
                           ▼                  ▼
                        REJECTED           CANCELLED
```

**每个状态返回的允许操作**（`allowedActions`）：

| 状态 | 允许操作 |
|---|---|
| DRAFT | edit / submit / cancel |
| SUBMITTED | approve / reject / cancel |
| APPROVED | receive（📦 收货入库） |
| RECEIVING | receive（📦 继续收货） |
| COMPLETED / REJECTED / CANCELLED | 无（仅查看） |

#### 输出：收货入库副作用

一次 `POST /api/purchase-orders/{id}/receive` 触发：

1. `purchase_order_lines.received_qty += 收货数量`
2. `inventory` 表：
   - 存在同 `(tenant_id, product_id, warehouse_id)` 记录 → `UPDATE qty += X`
   - 不存在 → `INSERT`
3. `inventory_transactions` 写入日志（`txn_type='RECEIPT'`, `qty_change=+X`, `ref_doc_type='purchase_order'`, `ref_doc_id=PO_ID`）
4. 检查所有明细 `received_qty >= qty` → 状态变 `COMPLETED`；否则 `RECEIVING`
5. 写 `audit_logs`（`action='PO_RECEIVE'`）

#### 异常
- 40404 采购单不存在
- 40009 当前状态不允许该操作（例：草稿单不能收货）
- 50000 库存写入失败（事务回滚）

### F.6 库存联动完整规则（v3.0 增强）

| 触发场景 | inventory 变化 | inventory_transactions 记录 |
|---|---|---|
| 采购收货入库 | `qty += 收货数量` | `RECEIPT`, `qty_change=+X`, ref=purchase_order |
| 销售出库发货（待开发） | `qty -= 出库数量` | `ISSUE`, `qty_change=-X`, ref=sales_out |
| 调拨（stock_moves） | 源仓 `qty -= X` · 目标仓 `qty += X` | 两条 `MOVE`, +/- X, ref=stock_move |
| 库存调整（adjustments） | `qty` 增/减 | `ADJUST`, ref=inventory_adjustment |
| 盘点差异 | 按差异更新 `qty` | `STOCKTAKE_ADJ`, ref=stocktake |

所有变动都写 `inventory_transactions` 分区表（按月）确保**审计可追溯**。
