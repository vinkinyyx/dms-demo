### v3.4.13 (2026-07-19) — 时区统一 + 列头筛选 + 库存调整/移动重构 + 日志与详情增强
- **时区统一（DateFmt）**：新增 `com.dms.common.util.DateFmt` 工具，把原生 SQL 查询得到的 `java.sql.Timestamp / OffsetDateTime / Date` 等时间对象统一按 `Asia/Shanghai` 格式化为字符串（与 JVM 默认时区无关）。根因：原生 SQL 路径用 `String.valueOf(Timestamp)` 绕过了 Jackson 的 `spring.jackson.time-zone` 配置，输出依赖 JVM 时区。改造覆盖 SupplierController/ProductPriceController/OperationLogController/BizDocListController/DashboardController/InventoryController 等 30+ 处；`BizDocDetailController.tupleToMap` 对以 `_at/_date/_time` 结尾的时间列统一格式化
- **库存调整重构（单仓盘盈/盘亏）**：`InventoryAdjustmentController` 重写为接受前端扁平结构 `{warehouseId, category:IN/OUT, type, stockStatus, remark, lines:[{productId,batchNo,serialNo,qty}]}`，直接完成库存增减 + 写 `adjustment_lines`（含 serial_no）+ `inventory_adjustment` 操作日志；新增 `GET /{id}/detail` 详情接口（join 仓库/产品名）。修复"打开查看/编辑报错"（此前无 GET-by-id 端点）与"新建无明细录入"
- **库存移动重构（跨仓）**：`StockMoveController` 重写为扁平结构 `{fromWarehouseId,toWarehouseId,stockStatus,remark,lines}`，源仓出库 + 目标仓入库两笔流水，写 `stock_move_lines`（含 serial_no）+ 操作日志；新增 `GET /{id}/detail`
- **主数据/取消操作日志补全**：DealerService/HospitalService/WarehouseService 注入 OperationLogService，create/update/deactivate 全部记录；ProductService.create/deactivate 补记录；OrderApprovalExecutionController 的 cancelSalesOut/cancelReceipt 补 `CANCEL` 日志
- **产品分类名称回填**：Product 实体加 `@Transient categoryName`，ProductService.get 注入 EntityManager 按 categoryId 查 product_categories 回填名称，详情查看态显示分类名
- **详情带出源单表头**：BizDocDetailController 的 salesOutDetail 按 `source_order_id` 查 orders + dealers 返回 `sourceOrder`；receiptDetail 按 `source_po_id` 查 purchase_orders + suppliers 返回 `sourcePo`
- **前端列表列头漏斗筛选**：`reloadList` 拆分为 fetch(`_raw`) + `_renderListTable` 渲染；工具栏"🔽 筛选"开关切换筛选行，`_setColFilter/_clearColFilters` 对当前页数据按列关键字过滤，保留原升/降序
- **前端单据编号链接**：菜单加 `detailable:true`，列表 code 列渲染为 `<a onclick="doView(id)">`；detail 路径白名单改为由 `m.detailable` 判定（含 stock-moves/inventory-adjustments）
- **前端明细录入编辑器**：`_renderFormBody` 非查看态且含 `type:lines` 字段时渲染可编辑明细表（`_addLineRow` 动态加行，picker 列异步拉 lookups 下拉）；`_submitFormTab` 收集 `.dms-form-grid` 顶层字段 + `#linesEditor` 明细行组装 `body.lines`
- **前端日期控件**：`_renderFormBody` 编辑态补 `f.type==='date'` → `<input type="date">` 分支（此前 date 字段落到纯文本 input）
- **前端序列号批量录入**：`openSerialBatchInput` 多行文本框弹窗替代逐个 prompt，支持 Excel 粘贴、实时计数、查重；receipts/sales-outs 的 statusActions 增加 `PARTIAL_RECEIVED/PARTIAL_SHIPPED` 的"继续收/发货"动作
- **后台菜单浅色**：admin.html 侧栏改 `#ffffff` 白底 + 深字，选中 `#e8f0fe` 浅蓝、`#1a56db` 高亮

### v3.4.12 (2026-07-19) — 收发货执行明细留痕 + 单据号序列 + 主数据操作日志 + 详情页名称化
- **收发货执行明细模型**：新增 `receipt_execution_lines` / `sales_out_execution_lines` 两张执行明细表，将"每一次收货/发货、每一个批次或序列号"作为独立记录留痕。设计要点：
  - 明细行汇总量（`receipt_lines.received_qty` / `sales_out_lines.shipped_qty`）只保存"累计已处理量"，真实的"第 N 次收了哪个批次多少件"下沉到执行明细表，解决历史 `batch_no` 被后一次覆盖导致详情页看不到多次记录的问题
  - `OrderApprovalExecutionController.executeReceipt/executeSalesOut` 每次执行前查 `MAX(seq_no)` 作为本次序号基线，逐条 `INSERT` 执行明细（`++seqBase`），记录 `operator_id` 与 `created_at`
  - `BizDocDetailController` 详情接口 join `products` + `users` 返回 `executionLines`，前端渲染"🔖 收/发货执行记录"表格（第 N 次 / 批次 / 序列号 / 数量 / 操作人 / 时间）
  - 明细行展示"待处理量 = 应收/发 − 已收/发"，首次收完后待收部分自动扣减
- **单据号序列生成器**：`DocNoGenerator` 由"时间戳尾数"重写为"数据库序列表原子自增"，格式统一 `PREFIX-YYYYMMDD-00001`
  - 新增 `doc_no_sequences(tenant_id, prefix, date_key, last_seq)` 表，`next(prefix)` 用 `INSERT ... ON CONFLICT (tenant_id,prefix,date_key) DO UPDATE SET last_seq = last_seq + 1 RETURNING last_seq` 原子自增，`@Transactional(REQUIRES_NEW)` 保证并发不撞号
  - 覆盖前缀：PO / RPO / RK / RRK / CK / RCK / SO / RSO；红字单用 R 前缀（RRK/RCK）
  - `PurchaseOrderController`、`AutoDocGenerator` 均改用 `DocNoGenerator.next()`
- **主数据编辑修复 + 操作日志**：
  - `ProductPriceController.update` 补齐此前遗漏的 `currency / effective_date / expire_date` 字段（`::DATE` cast + blankToNull/toBdOrNull 辅助）
  - `SupplierController.update` 改为全字段 `COALESCE(?, col)`，避免未传字段被误清空
  - suppliers / product_prices / products 的 create/update 统一接入 `OperationLogService`
- **产品分类下拉必填**：产品表单 `categoryId` 改为 `picker:{resource:'categories'}` 下拉 + required；V17 补充"监护设备/康复器械"分类测试数据，无分类产品补默认分类
- **详情页修复**：`_renderFormBody` 跳过 `type=lines` 字段（不再 [object Object]）；关联字段名称键修正（`dealerName` 而非 `dealerIdName`）；后端 `poDetail` join suppliers/warehouses 返回名称，supplierName 为空时兜底 supplierNameRef
- **序列号逐件维护**：收发货录入页对序列号产品拆多行逐个录入（收货手工批量输入、出库从合格库存多选），每序列号一条 `qty=1`，含 `_setSerials` / `openSerialMultiPicker` 辅助函数
- **后台菜单底色**：admin.html 侧栏 `#111827` → `#2b3a52`，深底文字/分组标题提亮

## 关键数据流

### 销售订单业务闭环 (v3.4+)

```
[销售] 创建订单(status=DRAFT, code=SO-xxx)
   │ 前端：选完经销商后，产品选择器只显示授权分类内的产品
   ↓
[销售] submit(status=SUBMITTED)
   ↓
[经理] approve(status=APPROVED, approved_at=now)
   │
   │ 触发 AutoDocGenerator.createSalesOutForOrder()
   ↓
[系统] 自动创建 sales_outs(code=CK-xxx, status=DRAFT, auto_created=true, source_order_id)
   │ 拷贝订单明细到 sales_out_lines
   ↓
[仓管] 进入销售出库页 → 看到 DRAFT 单
   │
   ├── execute → 弹窗填批次/序列号 → 校验库存 QUALIFIED → 扣减 → status=COMPLETED
   └── cancel-draft → status=CANCELLED（不动库存）
```

### 采购订单业务闭环

```
[采购] 创建采购订单(code=PO-xxx)
   ↓ approve → 自动创建 receipts(code=RK-xxx, DRAFT)
   ↓
[仓管] 进入收货入库页 → DRAFT 单
   ├── execute → **手工录入**批次/序列号（唯一手填场景）→ 增 PENDING → COMPLETED
   └── cancel-draft
```

### 分次收发货 + 执行明细留痕流程 (v3.4.10 → v3.4.12)

```
[仓管] 打开 RK/CK 草稿 → 收/发货录入子页
   │ 明细行显示：应收/发量 · 已收/发量 · 待处理量(=应收/发−已收/发) · 本次量
   │ 序列号产品 → 拆多行逐件录入(收货手输 / 出库从合格库存多选)，每序列号 qty=1
   ↓
[仓管] 提交本次 execute
   ↓
[后端] OrderApprovalExecutionController.executeReceipt / executeSalesOut
   │ 1) seqBase = SELECT COALESCE(MAX(seq_no),0) FROM *_execution_lines WHERE doc_id=?
   │ 2) 遍历每行的多条子录入(Map<lineId, List<sub>>)
   │    - inventoryOps.change 变更库存(收货增 PENDING / 出库扣 QUALIFIED)
   │    - INSERT *_execution_lines(seq_no=++seqBase, batch_no/serial_no, qty, operator_id, created_at)
   │ 3) received_qty / shipped_qty 累计
   │ 4) 全部收/发完 → COMPLETED；否则 → PARTIAL_RECEIVED / PARTIAL_SHIPPED
   ↓
[详情页] BizDocDetailController 返回 executionLines (join products + users)
   │ 渲染"🔖 收/发货执行记录"：第N次 · 批次 · 序列号 · 数量 · 操作人 · 时间
   └── 第一次收 ABC×5、第二次收 DEF×5 各留一条，不再互相覆盖
```

**设计取舍**：`receipt_lines` 仅保留一行汇总（received_qty 累计），若把多次多批次都塞进这一行，`batch_no` 只能存最后一次的值。因此把"每一笔"下沉到独立的执行明细表，是解决"分次收货详情看不到"的根因方案。

### 单据号生成流程 (v3.4.12)

```
[任意建单] DocNoGenerator.next("PO")
   ↓ @Transactional(REQUIRES_NEW)
[后端] INSERT INTO doc_no_sequences(tenant_id, prefix, date_key, last_seq)
       VALUES(:tid, 'PO', '20260719', 1)
       ON CONFLICT (tenant_id, prefix, date_key)
       DO UPDATE SET last_seq = doc_no_sequences.last_seq + 1
       RETURNING last_seq
   ↓
   String.format("%s-%s-%05d", prefix, date_key, seq)  →  PO-20260719-00001
```

- 原子 UPSERT + 独立事务保证并发下同租户同日单号连续、不撞号
- 前缀体系：PO / RPO / RK / RRK / CK / RCK / SO / RSO（R 前缀为红字单）


### 授权前置过滤流程

```
[前端] 销售订单表单 → dealerId 选择器绑定
              ↓
   产品明细行 → 点击"选择产品"
              ↓
   openPicker('products', callback, { dealerId: 3 })
              ↓
   GET /api/lookups/products?dealerId=3
              ↓
[后端] SELECT DISTINCT p.* FROM products p
       JOIN authorizations a ON a.dealer_id = 3
         AND a.status = 'active'
         AND CURRENT_DATE BETWEEN a.valid_from AND a.valid_to
         AND (a.category_ids = '' OR p.category_id IN split(a.category_ids))
              ↓
   只返回该经销商授权分类下的可下单产品
```

## 关键技术决策

| # | 决策 | 理由 |
|---|---|---|
| 1 | 前端 CSS 变量令牌 | 便于 M3 主题一键切换 |
| 2 | 用 native SQL 而非 JPA Sort | 修复 updatedAt 属性映射 500 |
| 3 | 用 CAST() 替代 ::text | Hibernate 会把 :xxx 当参数占位符 |
| 4 | 岗位与账号 UNIQUE 绑定 | 数据权限单一入口 |
| 5 | Auto 建单用 Native SQL | 避免 JPA 级联依赖 Entity 结构 |
| 6 | 单号前缀迁移 UPDATE 而非重建 | 保留 700 测试数据 |
| 7 | 授权过滤下沉到 Lookup | 前端组件单一，无需业务耦合 |
| 8 | 收发货执行明细独立成表 | 一行汇总放不下"多次多批次"，batch_no 会被覆盖；下沉留痕是根因解 |
| 9 | 单据号用 DB 序列表原子自增 | 取代时间戳尾数，保证 PREFIX-YYYYMMDD-连续号且并发不撞号 |
| 10 | 主数据 update 全字段 COALESCE | 部分字段未传时不误清空历史值 |
| 11 | 操作日志复用 audit_logs 表 | 不新增表，通过 resourceType/resourceId 统一时间轴查询 |
