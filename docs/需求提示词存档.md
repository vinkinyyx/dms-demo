6. 版本号、Flyway 版本、镜像 tag、测试通过数在各文档间保持一致，禁止臆造。

## 迭代 v3.4.14（据交付记录整理）

1. 产品编码修改后保存不生效 → 修复（update 补 setCode）。
2. 表单可选字段不应手输，应控件化（UDI 开关、数字框、日期控件、授权分类/终端多选下拉）。
3. 产品分类编码改不了 → 分类 update 补 setCode。
4. 产品价格关联产品显示 ID，产品页只有编码 → 关联列显示编码+名称。
5. 已收 2 次的收货单不能取消剩余 → cancelReceipt 放开 PARTIAL_RECEIVED。
6. 列表需要两种筛选（固定搜索区 + 表头漏斗），且跨全部数据（传后端 SpecUtil）。
7. 库存调整有明细却报错、库存移动要标注库存状态 → 库存调整改只读（一次性凭证），修更新时间时区。
8. 出库批次/序列号应从库存选择而非手输 → 输入框 readonly，仅按钮选取。

## 迭代 v3.4.15（用户原文）

> 明天我需要换一台电脑来继续修改 DMS ……（此为交接指令，另见下）

原始 8 条需求：

```
1. 所有单据/主数据，如果只是刚刚打开，或者输了一部分字段，切换了页面，
   再回来时会提醒用户单据无法打开，我想说地是如果没有保存或者提交地单据，
   不要记录和保存
2. 添加授权后，再次打开会提示报错；另外，列表页上体现地如经销商，授权产品分类，
   都体现地是ID而不是名称，我地要求是检查所有页面，不要出现只展示ID，而不展示名称，
   没有人可以看懂
3. 维护价格，保存时，提示内部系统错误
4. 库存移动中地仓库选择为什么和仓库管理中的仓库清单不一致
5. 库存移动的选择方式不够人性化，要求有几种方式，一种是像现在，正向地从仓库，物料，
   批次序列号来选则数据，二是让用户直接搜索批次号，或者序列号，或者物料，然后列出
   这个批次号/序列号/物料所有存在的仓库，库存状态等信息，给用户多选，选择后可以对
   这些数据进行批量移动，第三种是允许用户直接搜索关联收货入库单，将入库单中所有已经
   入库的明细带出来，调整这些明细库存
6. 销售订单，请确认可选择的物料是否遵守了经销商当前有有效授权的产品分类下的产品清单；
   另外，可用库存，请确认展示的库存是否是选择的物料在特定仓库中所有合格的库存，如果
   用户没有选仓库，可统计这个物料全部的合格库存
7. 仓库管理里的测试数据，不需要那么多，留2-3条即可
8. 在后台中，管理员可以调整菜单放在什么文件夹下一起展示
```

补充指令：「检查所有测试数据，删除不必要的测试数据，再创建一些必要的新的测试数据」

用户对澄清问题的最终选择：
- 第8条 → 完整后台可配置菜单分组
- 第5条 → 三种选择方式全做
- 第7条 → 既改 seed 又清理线上多余数据
- 第6条 → 修口径一致（下单校验也认分类授权）
